# インパクト評価フレームワーク

## 影響度レベルの定義

| レベル | 定義 | アクション | 推奨アクションの粒度 |
|--------|------|-----------|---------------------|
| 🔴 **High** | 今すぐ対応が必要。放置するとプロジェクトのブロッカーになる | issue 化を推奨 | 具体的な対応ステップ（「Package.swift の X を Y.Z に更新し、BreakingChange A を確認」） |
| 🟡 **Medium** | 近いうちに検討すべき。計画に組み込む価値がある | 次スプリントで議題に | 検討すべき論点と選択肢（「現在の〇〇設計を見直す価値がある。選択肢は A と B」） |
| 🟢 **Low** | 知っておくと良い。急ぎではないが方向性・設計に影響しうる | ナレッジとして記録 | 参考リンクと一言 |
| ⚪ **Info** | 直接の影響はないが、業界動向として把握しておくべき | 読み流し OK | 事実のみ（推奨アクションなし） |

---

## 評価軸の定義

| 評価軸 | 説明 | 判定の目安 |
|--------|------|-----------|
| **直接影響** | プロジェクトが使っている技術・ライブラリへの変更 | 依存ライブラリのメジャーアップデート、使用 API の deprecation、ビルドツールのブレイキングチェンジ |
| **間接影響** | プロジェクトの方針・設計判断に影響する変化 | 業界ベストプラクティスの変化、競合ツールの台頭、エコシステムのシフト |
| **機会** | 採用すればプロジェクトが改善される新技術・手法 | 新しいテストフレームワーク、ビルド高速化ツール、DX 改善ツール |
| **リスク** | 放置するとプロジェクトに悪影響が出る変化 | セキュリティ脆弱性、EOL アナウンス、ライセンス変更 |

---

## 影響度判定の具体例

### 直接影響 × 🔴 High の例
- 依存ライブラリのメジャーバージョンアップで、プロジェクトが使っている API が deprecated に
- プロジェクトが使っている言語/フレームワークのセキュリティ脆弱性が発見された
- CI/CD ツールの仕様変更でビルドが壊れる可能性がある

### 直接影響 × 🟡 Medium の例
- 依存ライブラリのマイナーアップデートで使いやすくなった機能がある
- 使っている API の deprecation アナウンスが出た（猶予期間あり）

### 機会 × 🟡 Medium の例
- プロジェクトが使っているフレームワークに、現在の設計を改善できる新機能が追加された
- プロジェクトのCI/CDを改善できる新しいツールが登場した

### 間接影響 × 🟢 Low の例
- 業界でよく使われている設計パターンの新しいベストプラクティスが議論されている
- 競合ツールが新機能をリリースした（直接使っていないが参考になる）

### ⚪ Info の例
- 関連分野のカンファレンスの開催アナウンス
- 業界全体のトレンドレポートの公開（直接影響なし）

## プロジェクトコンテキストとの突合チェックリスト

インパクト分析時に以下を確認する:

**依存関係の確認**:
- [ ] Package.swift / build.gradle 等に記載されているライブラリと一致するか
- [ ] バージョンが影響範囲内か（使用バージョン vs 変更が入ったバージョン）

**アーキテクチャへの影響**:
- [ ] AGENTS.md / CLAUDE.md に記載されているアーキテクチャ方針に影響するか
- [ ] 使っている設計パターン（MVVM, Clean Arch, etc.）に関係するか

**ツールチェーンへの影響**:
- [ ] .claude/skills/ に記載されているスキル・ワークフローに影響するか
- [ ] CI/CD や開発ツールに影響するか
