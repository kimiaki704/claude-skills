# 理論的背景

本スキルが取り入れているフレームワークの詳細リファレンス。名前を知らなくても使えるように設計しているが、裏付けとなる理論を理解することで判断の質が上がる。

## RACI（責任分担マトリクス）

タスクや領域ごとに4つの役割を明確にするフレームワーク。

| 役割 | 意味 | 本スキルでの対応 |
|------|------|-----------------|
| **R**esponsible（実行者） | 実際に手を動かす人 | テンプレートの「担当者」 |
| **A**ccountable（最終責任者） | タスクが完了するまで追う人。意思決定の場を設定し、合意を取り付け、実行を見届ける | テンプレートの「オーナー」 |
| **C**onsulted（相談先） | 壁打ちやレビューで意見を求める人 | テンプレートの「責任 > 相談先」 |
| **I**nformed（報告先） | 進捗や結果を共有すべき人 | テンプレートの「責任 > 共有先」 |

**重要な区別**: 意思決定はチーム全員で行ってよい。Accountable は「意思決定の内容」の責任者ではなく、「意思決定を駆動するタスク」の責任者。合意を取りに行く人 = その領域の Accountable。

**よくある失敗パターン**:
- Accountable が複数人 → 「誰かがやるだろう」になる。必ず 1 人に絞る
- Accountable = Responsible の兼任が常態化 → 起票者がすべて抱える構造になる。分離できないか検討する
- Informed を省略 → 完了したのに関係者が知らない状態になる

**適用上の注意**: RACI はタスク単位での役割定義に有効だが、組織全体の責任構造設計には向かない。また Accountable と Responsible を 1 人が兼任することが多い小規模チームでは、形式的な分離より実用性を優先してよい。

## チームトポロジー（Team Topologies）

Matthew Skelton / Manuel Pais による、チームの構造と責任範囲の設計理論（2019）。タスク分割時に以下の視点を使う。

| 概念 | タスク分割への適用 |
|------|------------------|
| **認知負荷（Cognitive Load）** | 1人が同時に抱えられるタスクの複雑さには限界がある。タスクが複数の専門領域にまたがる場合は分割すべきサイン |
| **チーム境界（Team Boundary）** | iOS / Android / バックエンド / デザインなど、担当の境界をまたぐタスクは境界ごとに分割するとオーナーシップが明確になる |
| **コミュニケーションパス** | 実行に必要な相談先が 3 人以上いる場合、タスクのスコープが広すぎる可能性がある |

**適用上の注意**: チームトポロジーはチーム設計全体の理論であり、「タスク分割の判断基準」はその一部の応用。チーム境界の「3人ルール」はあくまで目安で、文脈に応じて調整してよい。

## HIGH OUTPUT MANAGEMENT（タスク関連成熟度）

Andy Grove（Intel CEO）の著書 "High Output Management"（1983）が提唱する、部下の習熟度に応じた管理スタイルの調整理論。タスクの渡し方に応用する。

| 成熟度 | 渡し方 | 確認ポイントの密度 |
|--------|--------|------------------|
| 低（未経験） | 完了条件を細かく定義し、確認ポイントを多く設定 | 着手時・中間（1日ごと）・完了時 |
| 中（経験あり） | ゴールとスコープを明示し、進め方は任せる | 着手時・中間（1回）・完了時 |
| 高（熟練） | 背景だけ共有し、ゴール設定から任せる | 完了時のみ |

**起票者 ≠ 実行者を前提にする以上、タスクは成熟度「低〜中」でも動けるレベルで書く。** 熟練者が取った場合は不要な部分を読み飛ばせばよい。逆は成立しない。

**適用上の注意**: 成熟度の判定は起票者の観察に基づく目安。判定の参考として、タスク領域での実務経験が 3 ヶ月未満→低、半年〜2年→中、2年以上→高を目安にする。不明な場合は「中」として扱い、途中で密度を上げる。

## INVEST 原則

Bill Wake が提唱した、良いユーザーストーリー / タスクの品質基準（2003）。バックログリファインメントや分割後のタスク品質チェックに活用する。

| 原則 | 意味 | チェック方法 |
|------|------|-------------|
| **I**ndependent（独立） | 他タスクの完了を待たずに着手・完了できる | 依存関係が「ブロッカー」になっていないか |
| **N**egotiable（交渉可能） | スコープに柔軟性がある | 「やらないこと」で調整余地を残しているか |
| **V**aluable（価値がある） | 完了すること自体がプロジェクトに価値をもたらす | 背景セクションで価値が説明できるか |
| **E**stimable（見積り可能） | 担当者が工数を見積もれる | 完了条件が具体的か |
| **S**mall（小さい） | 1〜2 日で完了できるサイズ | 3 日超えるなら分割検討 |
| **T**estable（検証可能） | 完了を第三者が判定できる | 完了条件が客観的か |

**バックログリファインメントでの使い方**: スプリント計画前に Product Owner がバックログの各タスクを INVEST で評価し、基準を満たさないタスクをリファインメントセッションで修正する。すべての基準を完全に満たす必要はなく、Small と Independent が最優先。

**適用上の注意**: INVEST は元々ユーザーストーリー向けに設計されたが、エンジニアリングタスク全般に適用できる。ただし「Negotiable」は要件の交渉可能性を指すため、技術的に変更余地のないタスク（インフラ作業等）では適用が難しいケースがある。
