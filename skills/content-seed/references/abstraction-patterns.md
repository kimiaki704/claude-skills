# 社内情報の抽象化パターン

## 検出カテゴリと抽象化のガイドライン

### 組織名

| 検出パターン | 抽象化の方向性 | 例 |
|------------|--------------|---|
| 会社名 | 業種・規模・特徴で一般化 | 「ある決済系スタートアップ」「中規模 SaaS 企業」 |
| 部署・チーム名 | 機能・役割で表現 | 「モバイルアプリ開発チーム」「iOS/Android 各 2 名のチーム」 |
| プロジェクト名 | 用途・ドメインで表現 | 「クレジットカード管理アプリ」「BtoB 向け決済サービス」 |
| 社内コードネーム | 削除または「社内プロジェクト X」 | — |

### 人名

| 検出パターン | 抽象化の方向性 |
|------------|--------------|
| チームメンバーの名前 | 役割表記に変換（「PM」「テックリード」「iOS エンジニア」） |
| 上長・役員の名前 | 「マネージャー」「部長」等の役職に変換、または削除 |
| 取引先・ベンダーの担当者名 | 削除（「ベンダーの担当者」等に変換）|

### 数値

| 検出パターン | 抽象化の方向性 |
|------------|--------------|
| 売上・収益 | 削除、または「黒字化」「数億円規模」等の定性表現 |
| ユーザー数・DAU | オーダー表記（「数万ユーザー」「10 万 DAU 規模」）|
| 具体的な KPI | 目標の性質のみ記載（「高い可用性が求められる」）|
| コスト・予算 | 削除、または「コスト削減を優先」等の定性表現 |

### インフラ・社内システム

| 検出パターン | 抽象化の方向性 |
|------------|--------------|
| 社内ツール名（汎用ツール以外） | カテゴリで表現（「社内 CI 基盤」「社内ドキュメント管理システム」）|
| 内部 API の名称 | 「内部決済 API」「社内ユーザー管理 API」等に変換 |
| 本番環境のドメイン・IP | 削除 |
| 社内ネットワーク構成の詳細 | 概要のみ（「ゼロトラストアーキテクチャを採用」等）|

### 契約・ベンダー情報

| 検出パターン | 抽象化の方向性 |
|------------|--------------|
| 特定ベンダー名（公開情報でない場合） | カテゴリ表記（「決済代行サービス A」「クラウドプロバイダー」）|
| 契約の具体的な条件・金額 | 削除、または「有料プラン」「エンタープライズ契約」等に変換 |
| SLA の具体的な数値 | 「高い SLA が要求される」等の定性表現 |

## 抽象化レベルの判断基準

### 出力してよい情報

- 使用しているOSS・ライブラリ・フレームワークの名称（公開情報）
- 技術的な選択肢・トレードオフ（汎用的な内容）
- チームの規模感（人数のオーダー）
- 判断の観点・評価軸（汎用化された内容）
- 失敗や試行錯誤のエピソード（個人・組織が特定されない形で）

### 要注意・要確認

- 競合に対して優位性を示す可能性がある具体的な数値
- まだ公開されていない製品・機能に関する情報
- セキュリティ要件の詳細（脆弱性情報等）
- 特定の取引関係を示唆する情報

### 出力しない情報

- 個人を特定できる情報（氏名・メールアドレス等）
- 本番環境の構成情報（ドメイン・IP・認証情報）
- 未公開の財務情報
- 守秘義務のある取引条件の詳細

## 抽象化後のチェックリスト

スキルが抽象化案を提示した後、ユーザーに以下の確認を促す：

- [ ] 組織名・プロジェクト名が特定できる表現が残っていないか
- [ ] 個人が特定できる記述がないか
- [ ] 競合他社に対して情報優位性を示す可能性のある数値がないか
- [ ] 社内固有の文脈に依存しすぎて、汎用的な内容として読めるか
- [ ] 全体を読んで「これを公開しても問題ない」と自信を持って言えるか
